# Persistent Thumbnail Cache Global Plan

> Scope note: this is a planning document only. Do not modify production code until the plan is reviewed and execution is explicitly started.

## Goal

Make generated grid thumbnails persistent across app restarts by storing thumbnail blobs in SQLite, while keeping the current user-visible page and selected preview path faster than any background cache generation.

The cache must not become another blocking preload pipeline. Default behavior remains on-demand: only thumbnails that are generated by normal browsing are persisted. Users may manually request full thumbnail-cache generation from settings, and that work must run at the lowest priority.

## Current Baseline

- Core already has `thumb_blobs` for persistent thumbnail bytes.
- Core already serves `/media/{fileId}/thumbnail/blob` from DB blobs.
- Frontend already has an in-memory object URL thumbnail cache, currently exposed as a thumbnail cache size setting.
- Thumbnail workers already support selected, visible, ahead, and background priority concepts.
- Scan/upsert already has source fingerprint logic, but the current fingerprint includes path-like identity, so move/rename reuse needs redesign.

## Definitions

- Persistent thumbnail cache: SQLite `thumb_blobs` rows stored on disk and surviving restart.
- Memory thumbnail cache: frontend object URL/blob cache used to avoid repeated HTTP blob reads during the current renderer session.
- Foreground thumbnail generation: selected image, current visible viewport, and near-ahead thumbnails.
- Bulk thumbnail cache generation: user-requested or auto-scheduled generation for missing/stale thumbnails outside the immediate viewport.
- Content identity: the strongest available proof that a file is the same bytes as before. Prefer file system `file_key` plus `size` and `mtime`; if `file_key` is unavailable, use a conservative path-bound identity instead of cross-path reuse.

## Product Behavior

### Default Behavior

- Do not generate every thumbnail by default.
- Persist any thumbnail that is generated by normal browsing.
- When reopening a cached folder, the grid should use DB blobs and then populate the frontend memory cache.
- The visible viewport, clicked/selected image, and ahead-of-scroll thumbnails remain the fastest path.

### Manual Persistent Cache Controls

Settings should expose:

- Generate current folder thumbnails.
- Generate current file tree thumbnails.
- Generate all thumbnails.
- Retry failed/stale thumbnails.
- Clear persistent thumbnail cache.
- Show cache stats: cached count, missing/stale count, failed count, total DB blob bytes, active bulk tasks.

### Automatic Refresh

Settings should expose an opt-in auto-refresh option:

- When opening a folder, enqueue missing/stale thumbnails for that folder at bulk priority.
- When browsing with subfolder contents enabled, enqueue missing/stale thumbnails for that file tree at bulk priority.
- This trigger must be debounced by `(rootId, folderId, includeDescendants)` so rapid folder switching does not enqueue duplicate work.

## Data Consistency Rules

### Reuse Rules

- Reuse thumbnails across rename/move only when content identity proves the file is unchanged.
- A reliable match requires `file_key`, `size`, `mtime`, and profile to match.
- If `file_key` is unavailable, do not cross-path reuse. Use a conservative fallback identity that includes path/folder/name.
- Profile remains part of the identity because future thumbnail profiles may differ.

### Invalidation Rules

- If file content changes, reset thumbnail state and delete or ignore stale DB blob.
- If source file is missing, deleted, or no longer active, do not serve the blob and schedule cleanup.
- If a thumbnail task finishes after the file changed, publish must fail via source-fingerprint/attempt guard.
- A failed thumbnail for a now-changed source must be retryable.
- A failed thumbnail for the same unchanged bad source may remain terminal until manual retry.

### Cleanup Rules

- Deleting a file or root should remove associated blobs through FK cascade where possible.
- Marking files `missing` or `deleted` should exclude them from all cache candidate queries.
- Add a cleanup path to remove inactive orphan blobs and stale rows.
- Manual clear should support at least all-cache clearing; scoped clear by root/folder is preferred if it stays simple.

## Priority And Scheduling Model

Priority order must be:

1. Selected thumbnail and selected preview support.
2. Visible viewport thumbnails.
3. Ahead/preload thumbnails near the viewport.
4. Normal background thumbnail work requested by existing browsing paths.
5. Bulk persistent-cache generation.

Important constraints:

- Bulk work must never consume selected or visible reserved capacity.
- Bulk pending tasks must be promotable when the same file becomes visible or selected.
- Running bulk tasks must yield at existing thumbnail checkpoints when higher-priority thumbnail work appears.
- Bulk generation must use small DB transactions and wake workers incrementally, not enqueue or process million-file batches under one long lock.
- Existing visible-priority demotion on folder switch and scroll must continue to work.

## Storage Model

Use existing main SQLite DB first.

Reasoning:

- Existing `thumb_blobs` already stores persistent WebP bytes.
- Existing routes already return DB blobs.
- Existing migrations and tests already know about `thumb_blobs`.
- This minimizes migration risk and avoids a second DB consistency layer.

Known risk:

- Million-scale caches can grow the main DB and WAL significantly.
- Pressure tests must record DB size, WAL size, write latency, and visible-thumbnail latency while bulk generation is active.
- If this becomes the bottleneck, a later plan can split thumbnail blobs into a dedicated cache DB.

## API Plan

Add contract-first endpoints:

- `GET /api/thumbnails/cache/stats`
- `POST /api/tasks/thumbnail-cache`
- `POST /api/thumbnails/cache/clear`

Request concepts:

- `rootId?: number`
- `folderId?: number`
- `includeDescendants?: boolean`
- `refreshMode: "missingOnly" | "staleOrMissing" | "retryFailedAndStale"`
- `limit?: number` for bounded enqueue batches

Response concepts:

- accepted task/enqueue count
- cached count
- pending candidate count
- failed count
- stale count
- total byte size
- active bulk task count

Client rules:

- Update OpenAPI first.
- Update generated contract/client/types.
- `apps/web` must call these through `@megle/core-client`, not raw `fetch`.

## UI Plan

Settings must distinguish:

- Thumbnail memory cache: frontend object URL cache, default currently 5GB.
- Persistent thumbnail cache: SQLite DB thumbnail cache stored on disk.

Settings copy must explain:

- Memory cache improves current-session switching.
- Persistent cache improves reopening folders and app restarts.
- Full generation is optional and lowest priority.

Controls:

- Memory cache size slider remains.
- Persistent cache stats section.
- Auto-refresh toggle.
- Generate current folder.
- Generate current tree.
- Generate all.
- Retry failed/stale.
- Clear persistent cache.

UI constraints:

- Do not add heavy polling that causes settings or library jank.
- Stats refresh should be manual plus lightweight periodic refresh only while settings is open or a bulk job is active.
- Do not show persistent cache generation as if it is blocking foreground browsing.

## Implementation Phases

### Phase 1: Core Identity And Cleanup

- Redesign thumbnail source fingerprint to be content-first.
- Add tests for rename/move reuse and modified-file invalidation.
- Add DB cleanup helpers for stale and inactive blobs.
- Ensure delete/missing paths exclude and clean thumbnails.

### Phase 2: Bulk Cache Priority

- Add an internal priority lower than existing background.
- Add candidate queries for missing/stale/failed thumbnails.
- Add bounded enqueue helpers.
- Ensure pending bulk tasks promote when foreground requests arrive.
- Ensure running bulk tasks yield to higher priorities.

### Phase 3: API And Core Client

- Add OpenAPI endpoints and DTOs.
- Add Core routes and client methods.
- Add contract guards and client tests.
- Add API tests for stats, enqueue, clear, and retry modes.

### Phase 4: Settings And Auto-Refresh

- Rename memory cache UI.
- Add persistent cache controls and stats.
- Add auto-refresh preference.
- Trigger low-priority scoped enqueue when opening folders or trees.
- Debounce scope-triggered enqueue.

### Phase 5: Pressure Testing And Tuning

- Build a repeatable desktop pressure harness.
- Test cold cache, hot cache, bulk generation active, deep scroll, folder switching, and preview open/close.
- Tune worker permits, enqueue batch size, checkpoint frequency, and DB write batch size based on measured bottlenecks.

## Important Implementation Notes

- Do not weaken `check:core-api`, `check:core-client`, `check:web`, `check:ui-design`, or schema checks.
- Do not add direct filesystem access in `apps/web`.
- Do not add raw `fetch` in UI feature code.
- Do not make full-cache generation default-on.
- Do not block folder browsing on cache stats or cache enqueue.
- Do not load all media IDs into frontend memory for full-cache generation.
- Do not hold SQLite write transactions while scanning a million-file candidate set.
- Do not let bulk generation starve visible/selected thumbnail requests.
- Do not reuse a thumbnail after file modification.
- Do not serve a DB blob if its current source fingerprint is stale.
- Do not show stale thumbnail blobs for deleted/missing files.

## Test Plan

### Unit Tests

Rust:

- Content fingerprint remains stable for same `file_key + size + mtime` across rename/move.
- Content fingerprint changes for changed size.
- Content fingerprint changes for changed mtime.
- Content fingerprint changes for changed file_key.
- Missing `file_key` fallback does not allow cross-path reuse.
- Stale source resets thumbnail state.
- Deleted/missing files are excluded from cache candidates.
- Inactive blob cleanup removes orphan/inactive blobs.
- Bulk priority is lower than background.
- Pending bulk task promotes to visible/selected.
- Running bulk task yields when higher-priority work appears.
- Stale task attempt cannot publish after source change.

TypeScript/client:

- Client methods serialize cache scope and refresh mode.
- Client forwards abort signal.
- Contract guard rejects direct/raw UI fetch path.
- Settings preferences normalize persistent-cache options.

### Integration Tests

Core API:

- Stats returns correct counts and byte totals.
- Enqueue current folder only schedules active missing/stale files.
- Enqueue tree respects `includeDescendants`.
- Clear cache deletes blobs and resets thumbnail states.
- Retry failed mode requeues failed thumbnails only when requested.

Schema:

- Migrations apply cleanly from empty DB.
- Migrations recover from partial application where required.
- Required indexes exist for candidate queries and stats.

### Real Desktop Pressure Tests

Hot-cache scenarios:

- Restart app after cache generation.
- Reopen previously cached folders.

Metrics:

## Acceptance Criteria

- Normal browsing persists generated thumbnails without a manual full-cache operation.
- Full-cache generation is opt-in and always lower priority than every foreground path.
- Visible viewport thumbnails refresh faster than bulk thumbnails.
- Selected/clicked image work jumps ahead of all cache generation.
- Moving or renaming an unchanged file reuses the thumbnail when reliable content identity matches.
- Modifying a file invalidates old thumbnail data.
- Missing/deleted files do not keep serving stale thumbnails and can be cleaned.
- Hot-cache folder revisit is faster than cold-cache browsing.
- No UI freeze or folder-switch blocking while bulk generation is active.
- All contract, web, schema, and focused Rust thumbnail tests pass.

## Open Decisions Before Coding

- Whether scoped cache clear by folder/tree is required in the first implementation, or all-cache clear is enough.
- Default auto-refresh setting: recommended default is off.
- Bulk enqueue batch size: initial recommendation is bounded chunks of 256 candidates, then tune with pressure tests.
- Persistent cache stats refresh cadence: recommended manual refresh plus low-frequency refresh while settings is visible.
