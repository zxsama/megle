# Component Library Review

Updated: 2026-05-16

## Scope

This review only covers component-level open source libraries. It does not evaluate existing photo management applications.

Megle needs:

- Windows-first desktop app.
- Existing-folder indexing only.
- Millions of images/videos.
- Fast thumbnail browsing and no-stutter preview switching.
- Tags/ratings stored in Megle database.
- Real file rename/move/delete.
- Future Web/Docker deployment.
- Plugin-ready architecture.

## Final Component Stack

```text
Desktop Shell      Electron
UI                 React + TypeScript + Vite
UI Components      Radix UI + Tailwind CSS + Lucide
Virtual Grid       TanStack Virtual
Server State       TanStack Query
Local UI State     Zustand
Core Runtime       Rust + Tokio + Axum
Database           SQLite WAL + FTS5 + rusqlite
Filesystem         walkdir/jwalk + notify + windows-rs
File Operations    windows-rs + trash-rs
Image Pipeline     libvips primary, image/fast_image_resize fallback
Video Pipeline     FFmpeg sidecar
Metadata           exif-rs first, ExifTool sidecar optional
Search             SQLite FTS5 first, Tantivy later
Hashing            BLAKE3, background only
Plugins            Process plugins first, Wasmtime/Extism later
Logging            tracing
Parallel CPU       rayon
```

## UI And Desktop

### Electron

Use Electron for the first Windows desktop release.

Why:

- Mature desktop shell.
- Chromium rendering is predictable for dense media grids.
- Easier to build an Eagle-like interaction model than with native UI from scratch.
- The UI can be reused for future Web/Docker.

Risk:

- Larger package and higher memory use than Tauri.
- Must avoid putting indexing, thumbnail generation, and decoding into Electron main or renderer.

Decision:

- Electron is the shell only.
- All heavy work goes through `megle-core.exe`.
- Tauri remains a future shell replacement option, not the first implementation.

### React + TypeScript + Vite

Use React for the UI and Vite for the build.

Why:

- Fast iteration.
- Works for desktop and future Web UI.
- Large ecosystem for virtualized grids, accessible primitives, and data fetching.

### TanStack Virtual

Use TanStack Virtual for the media grid and long lists.

Rules:

- Never render all items.
- Use fixed or predictable item dimensions.
- Keep overscan bounded.
- Use cursor pagination from Core.
- Do not let image load events change grid layout.

Fallback:

- If DOM virtualization cannot hit performance goals with 1M test rows, prototype a Canvas/WebGL grid later. Do not start there.

### Radix UI + Tailwind CSS + Lucide

Use Radix UI primitives for accessible menus, dialogs, popovers, tabs, tooltips, sliders, and context menus.

Use Tailwind CSS for styling tokens and layout utilities.

Use Lucide for icons.

Liquid glass visual rule:

- Apply blur/translucency to toolbar, sidebar, popovers, and settings.
- Do not place large `backdrop-filter` layers behind the media grid.
- Add a performance setting that can reduce blur, shadows, and animation.

### TanStack Query + Zustand

Use TanStack Query for server state from Core:

- directory children
- paged media lists
- item metadata
- thumbnail URL state
- background task status

Use Zustand only for local UI state:

- selected item
- preview mode
- panel widths
- grid zoom
- active filters before submit

Do not duplicate the database state in the frontend store.

## Core Service

### Rust + Tokio + Axum

Use a separate Rust process as the Core Service.

Responsibilities:

- root management
- directory scanning
- database access
- search
- thumbnails
- metadata extraction
- file watching
- real file operations
- plugin host

Use Tokio for async IO and Axum for HTTP API. The same API shape can serve the future Web/Docker version.

Desktop security:

- Bind only to localhost or use named pipe.
- Require a random session token generated by the desktop shell.
- Never expose arbitrary filesystem access without registered roots.

### rayon

Use rayon for CPU-heavy parallel work:

- image resizing
- hash computation
- metadata post-processing

Keep rayon CPU pools separate from Tokio IO work so CPU tasks do not starve API requests.

### tracing

Use tracing for structured logs:

- scan jobs
- thumbnail jobs
- decoder failures
- file operations
- plugin lifecycle

## Database And Search

### SQLite WAL + rusqlite

Use SQLite in WAL mode with rusqlite.

Rules:

- Single writer queue.
- Multiple read connections.
- Batch writes in transactions.
- Use keyset pagination.
- Avoid large offset pagination.
- Drive hot browse queries from the final sort index.
- Avoid join plans that filter millions of rows and then create a temporary sort tree.
- Run `ANALYZE` after large imports.
- Store folders and files separately to avoid repeating full paths in every row.

SQLite is sufficient for millions of media metadata rows if queries are disciplined.

### FTS5

Use SQLite FTS5 for first-stage text search:

- file names
- relative paths
- tags
- notes

Tantivy can be introduced later if full-text search becomes too complex or needs independent indexing.

### BLAKE3

Use BLAKE3 for content hashing, but only in background or on demand.

Do not hash every file during initial scan. Initial identity should use:

- root id
- folder id
- file name
- file size
- mtime
- platform file id when available

## File System

### walkdir / jwalk

Use walkdir first for reliable recursive traversal. Use jwalk if profiling shows directory traversal is a bottleneck.

Rules:

- Initial scan should write file rows quickly before thumbnails are generated.
- UI should become browsable while the scan continues.
- Directory tree must be lazy loaded.

### notify

Use notify for filesystem events.

Important:

- Watcher events are hints, not truth.
- Event overflow, wake-from-sleep, network drives, and removable drives require local rescan.
- A watcher event should enqueue verification work, not directly mutate final state blindly.

### windows-rs

Use windows-rs for Windows-specific APIs:

- file attributes
- file identity
- shell integration
- recycle bin behavior when trash-rs is insufficient
- future NTFS USN Journal exploration

### trash-rs

Use trash-rs for default delete-to-recycle-bin behavior if it passes Windows tests.

Fallback to windows-rs Shell APIs for edge cases.

Deletion policy:

- Default delete goes to recycle bin.
- Permanent delete requires explicit advanced confirmation.
- Every operation writes to `file_operations`.

## Media Pipeline

### libvips

Use libvips as the primary image thumbnail engine.

Why:

- Designed for low memory, streaming/demand-driven image processing.
- Better fit than loading giant images fully into memory.

Integration choice:

- Prefer a Rust-compatible binding or controlled sidecar worker.
- Keep decoding outside the UI process.
- Generated thumbnails should be encoded as WebP by default.
- AVIF can be revisited later for cold-cache storage, but it is not the first-release thumbnail format.

### image + fast_image_resize

Use `image` and `fast_image_resize` as a fallback path for common simple formats or already-decoded pixels.

Do not make `image` the only thumbnail engine for large/complex formats.

### FFmpeg

Use FFmpeg sidecar/subprocess for:

- video metadata
- poster frames
- optional previews
- future range streaming in Web/Docker

Rules:

- Limit concurrency.
- Put video jobs behind interactive image thumbnail work.
- Cache poster frames.
- Carefully choose LGPL-compatible builds unless GPL licensing is acceptable.

### Exif And Long-Tail Formats

Use exif-rs first for basic EXIF.

Use ExifTool as optional sidecar only when broad metadata coverage is required.

Long-tail formats should be plugin based:

- RAW through LibRaw plugin.
- HEIC/AVIF through libheif plugin.
- PSD/TIFF/rare formats through isolated ImageMagick plugin.

Do not let risky decoders run inside the Core process if they can crash or hang.

## Plugin Runtime

### Process Plugins First

Use process-based plugins first.

Benefits:

- Lowest implementation risk.
- Language independent.
- Crash isolation.
- Works for native decoders.

Plugin types:

- decoder
- metadata
- action
- import provider

The browser extension should later be an import provider plugin, not a built-in first-release feature.

### Wasmtime / Extism Later

Use Wasmtime or Extism later for sandboxed logic plugins where WASM fits.

Do not use WASM first for media decoders; native dependencies make that path harder.

## Libraries To Avoid In The First Release

Avoid as first-release core dependencies:

- PostgreSQL: too much deployment cost for local desktop first.
- RocksDB: query flexibility and tooling cost are worse than SQLite for this app.
- DuckDB: excellent analytics engine, not the primary app database.
- Elasticsearch/OpenSearch: too heavy for local desktop.
- ImageMagick as the only image engine: broad format support but higher security/performance risk.
- Canvas/WebGL custom grid as the first UI: too much complexity before TanStack Virtual is proven insufficient.
- Full AI/vector stack: should be plugin/phase later.

## Prototype Gates

Before full app implementation, build these gates:

1. SQLite gate
   - Generate 1M and 5M file/media rows.
   - Test folder query, filters, sorting, keyset pagination.
   - Target common folder first-page query under 100ms after warmup.

2. Virtual grid gate
   - Render 1M fake media entries through paged Core API.
   - Keep scroll smooth with stable item sizes.
   - Verify glass UI effects do not affect grid FPS.

3. Thumbnail gate
   - Generate tiny/grid/retina thumbnails for real images.
   - Generate poster frames for videos through FFmpeg.
   - Verify current viewport jobs preempt background jobs.

4. Preview gate
   - Switch selection 100 times rapidly.
   - UI selection update target under 50ms.
   - Cached preview target under 150ms.

5. File operation gate
   - Rename/move/delete real files in test roots.
   - Simulate conflict and failure.
   - Confirm database and filesystem stay consistent.
