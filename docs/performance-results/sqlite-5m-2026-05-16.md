# SQLite 5M Benchmark

Date: 2026-05-16

## Purpose

Validate that the SQLite metadata index design scales beyond 1 million rows and remains viable at 5 million media files.

This is still a metadata benchmark, not a thumbnail or filesystem scan benchmark.

## Environment

- OS: Windows 11 `10.0.26200`
- Python: `3.13.3`
- SQLite: `3.49.1`
- Visible CPU threads: `32`
- Drive: `D:\`, NTFS, about `296 GB` free before the run

## Dataset

- Roots: `4`
- Folders: `10,000`
- Files: `5,000,000`
- Media rows: `5,000,000`
- User metadata rows: `5,000,000`
- FTS rows: `5,000,000`
- Batch size: `10,000`
- Initial database size after checkpoint: `2,120.04 MB`

## Query Plan Check

Representative plans:

- `folder_first_page` uses `idx_files_folder_name` and primary-key lookup into `media`.
- `recent_images` scans `idx_files_mtime` and verifies `media` by primary key.
- `fts_tag_search` uses the FTS5 virtual table index.

The previous 1M finding still applies: browse queries must be driven by the final sort index.

## WAL Concurrency Test

Purpose:

Simulate foreground browsing while a background worker writes thumbnail records into `thumbs`.

### Scenario A

- Readers: `4`
- Background writes: `100,000`
- Write batch size: `1,000`
- Elapsed: `351.840 ms`
- Write throughput: `284,220 rows/sec`

| Operation | p50 | p95 | p99 |
| --- | ---: | ---: | ---: |
| Folder first page | `3.450 ms` | `4.549 ms` | `5.178 ms` |
| Recent images | `2.327 ms` | `3.482 ms` | `3.865 ms` |
| FTS tag search | `0.938 ms` | `1.480 ms` | `2.283 ms` |
| Write batch | `2.825 ms` | `5.412 ms` | `8.125 ms` |

### Scenario B

- Readers: `8`
- Background writes: `500,000`
- Write batch size: `1,000`
- Elapsed: `8,473.213 ms`
- Write throughput: `59,009 rows/sec`

| Operation | p50 | p95 | p99 |
| --- | ---: | ---: | ---: |
| Folder first page | `10.938 ms` | `14.324 ms` | `16.223 ms` |
| Recent images | `8.585 ms` | `12.010 ms` | `13.619 ms` |
| FTS tag search | `5.439 ms` | `8.375 ms` | `9.677 ms` |
| Write batch | `18.044 ms` | `28.137 ms` | `31.859 ms` |

## Concurrency Result

SQLite WAL with short write transactions is viable for foreground browsing plus background thumbnail/index writes.

Important implementation rules:

- Use one writer queue.
- Keep write transactions short.
- Batch writes, but do not make batches unbounded.
- Keep read connections separate from the writer connection.
- Prioritize foreground reads over background write throughput.
- Add backpressure to thumbnail/index workers.

## Paged API Serialization Test

Purpose:

Approximate the Core-to-UI data path by fetching random folder pages and serializing them to JSON.

Scenario:

- Database: 5M media rows.
- Iterations: `2,000`
- Page size: `200`
- Random folders: `1..10,000`
- Response includes file fields, media fields, user metadata, and thumbnail URL.

| Metric | Value |
| --- | ---: |
| Query + map + JSON p50 | `0.866 ms` |
| Query + map + JSON p95 | `1.831 ms` |
| Query + map + JSON p99 | `2.617 ms` |
| Max | `3.596 ms` |
| Payload p50 | `42,194 bytes` |

Result:

- Paged response serialization is not a current bottleneck.
- A 200-item media page is a reasonable first API page size.
- The UI should still request only the viewport window plus bounded overscan.

## Result

The SQLite metadata gate passes at 5M rows.

This validates the chosen database direction for:

- folder browsing
- keyset pagination
- indexed filters
- FTS5 search
- concurrent foreground reads and background thumbnail writes

Remaining gates:

- Real filesystem scanning throughput.
- Thumbnail generation throughput.
- UI virtual grid scrolling.
- Preview switching and image decode prefetch.
- File operation consistency.
