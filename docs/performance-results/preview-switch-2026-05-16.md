# Preview Switch Benchmark

Date: 2026-05-16

## Purpose

Validate the preview switching path:

- selected item state changes immediately
- cached WebP preview displays on the next frame
- nearby items are preloaded
- rapid key-repeat style changes do not commit the wrong final item

This benchmark uses the WebP thumbnails generated by the thumbnail pipeline test.

## Stack

- React
- Vite
- Playwright Chromium
- WebP preview assets: `128`
- Viewport: `1440x900`
- Preview preload radius: `12`

## Method

Three modes:

1. `cached`
   - Preload all 128 WebP preview assets.
   - Switch selection 1000 times.
   - Await one animation frame per switch.

2. `prefetch`
   - Use neighbor prefetch radius 12.
   - Switch sequentially 1000 times.
   - Await one animation frame per switch.

3. `burst`
   - 200 bursts.
   - 20 selection changes per burst.
   - Await one animation frame per burst.
   - Only final selected id per burst must commit.

## Result

The preview switch gate passes for cached WebP previews.

The measured switch latency is one animation frame, as expected:

```text
p99 ≈ 17.6 ms at 60Hz
```

WebP decode cost is low once cached:

```text
decode p99 ≈ 0.8-1.2 ms
```

Rapid burst updates commit the correct final selection with no mismatches.

## Product Rules

- Selection state must update immediately and must not wait for original image decode.
- The previewer should display cached WebP first.
- Original image decode must be asynchronous and cancellable.
- Neighbor prefetch should default around 12 items, then become adaptive.
- Rapid key-repeat should coalesce stale intermediate work and only keep the newest selected item interactive.

## Caveats

- This test uses generated 320px WebP thumbnails, not full-size preview images.
- It does not test decoding 1600px preview cache images.
- It does not test RAW/PSD/HEIC plugin decode.
- It does not test memory pressure with thousands of unique large previews.
