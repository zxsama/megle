# File Operation Consistency Benchmark

Date: 2026-05-16

## Purpose

Validate real file operation ordering and database consistency for:

- rename
- move
- soft delete
- conflict failure logging

This benchmark uses a sandbox test root under `tools/bench/fs-scan/results/`.

## Method

Dataset:

- Files: `1,000`
- Initial folder: `inbox`
- Target folder: `sorted`
- Soft-delete folder: `.trash`

Operations:

- Rename `300` files.
- Move `400` files.
- Soft-delete `200` files.
- Attempt one intentional rename conflict.

Operation ordering:

1. Validate target path.
2. Execute filesystem operation.
3. Update SQLite state in a transaction.
4. Write `file_operations` log.

Conflict behavior:

- If target exists, operation fails.
- The target file must not be overwritten.
- The source file/database row must stay unchanged.
- A failed operation log row must be written.

## Decision

The basic operation order is correct:

- filesystem first
- database second
- operation log always
- no overwrite by default

Implementation rules:

- Validate target path before rename/move/delete.
- Never overwrite an existing target unless the user explicitly chooses overwrite.
- Failed operations must not mutate the file row.
- Operation logs are required for both success and failure.
- Delete in the real app should use the Windows Recycle Bin, not a local `.trash` folder. This benchmark uses `.trash` only as a portable stand-in.

## Remaining Work
